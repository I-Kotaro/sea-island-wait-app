# 水族館イベント待ち時間アプリ (ポートフォリオ)

Spring Boot と Tailwind CSS を用いて、
**水族館イベントの待ち時間表示・応募管理を想定したWebアプリ**を開発しました。<br>
イベントへの応募機能や待ち組数・待ち時間を可視化、ユーザーが応募中のイベントを確認できるチケットページ、<br>
イベントに応募中は別イベントへ応募できないよう、ダブルブッキングを防止する機能なども実装し、<br>
ユーザーがスムーズに体験できることを目的としてロジック設計を行っています。<br>

## アプリについて

本ポートフォリオで使用しているイベント風景の画像は、生成AIツール **Whisk** により作成しています。

## 苦労点・気付き・学び

### Git運用（ブランチ・プルリクエスト）

本ポートフォリオでは、少しでも実務に近づけようと GitHub でブランチ運用とプルリクエストの流れを一人で実践しました。<br>
当初は、Controller,Serviceなど層ごとにブランチを切るものだと考えており、必要以上にブランチを細かく切ったのが原因で途中からコンフリクトが多発する事態になり、<br>
ブランチ運用の考え方に誤りがあったことに気づきました。<br>

以降は、実際に開発を進める中で、ブランチは**機能単位で切るのが適切である**と学び、画面作成、機能実装、UI改善など、<br>
まとまりのある単位でブランチを作成し、プルリクエスト単位で変更を管理することで、変更内容が把握しやすくなり、効率的に開発を進められるようになりました。<br>

ポートフォリオ開発段階での失敗を経て、ブランチの粒度設計の重要性を理解することができ、実務においても誤ったブランチ運用による混乱を避け、<br>
スムーズに開発を進められると考えています。<br>

### データ設計（カウント方法）

アプリ構想段階では、イベントの応募数などの値について、テーブルにカウント用のカラムを持たせて管理する設計を考えていました。<br>

しかし、実際に設計を進める中で、応募やキャンセルなどにより頻繁に変動する値をカラムとして保持すると、更新漏れや不整合が発生するリスクがあると知り、<br>
応募数のような可変データについてはカラムで管理するのではなく、応募テーブルのレコード数を count で集計することで取得する設計に変更しました。

この経験から、データの整合性を保つためには、値を保持するか動的に算出するかを適切に判断することが重要であると学びました。
